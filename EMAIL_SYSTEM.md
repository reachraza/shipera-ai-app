# Shipera.AI Email System Documentation

This document explains how the email architecture works in the Shipera.AI application, covering both outbound (sending invites) and inbound (receiving replies) flows using SMTP and the Gmail API.

---

## 1. Architecture Overview

Shipera.AI handles emails in two directions:
1.  **Outbound**: Sending RFP invitations to carriers via SMTP (Nodemailer).
2.  **Inbound**: Automatically fetching and verifying replies to those invitations via the Gmail API.

---

## 2. Outbound Flow (Sending)

When a shipper sends an RFP invitation, the application uses **Nodemailer** with SMTP credentials (configured in `.env.local`).

### Process:
1.  **Request**: The frontend calls `/api/email/send-invites`.
2.  **SMTP Send**: The server uses `nodemailer` to send the email.
3.  **Tracking**: The unique `Message-ID` generated by the mail server is captured.
4.  **Storage**: This `Message-ID` is saved in the `rfp_invites` table under the `last_message_id` column. This is the "fingerprint" used to recognize replies later.

**Key Files:**
- `src/services/emailService.ts`: Templates and sending logic.
- `src/app/api/email/send-invites/route.ts`: API endpoint for sending.

---

## 3. Inbound Flow (Receiving & Verifying)

Instead of traditional Webhooks (which require a public URL), we use the **Gmail API** to "pull" emails from the authorized Gmail account.

### Authentication:
The app uses **OAuth2** with a **Refresh Token**. This allows the server to check for emails 24/7 without requiring a user to be logged in.
- **Client ID & Secret**: Obtained from Google Cloud Console.
- **Refresh Token**: Stored in `GOOGLE_REFRESH_TOKEN` in `.env.local`.

### Process:
1.  **Fetch**: The `/api/email/fetch-gmail` endpoint is triggered.
2.  **List**: It fetches the latest unread messages from the authenticated Gmail account.
3.  **Verify**: For each email, it checks the `In-Reply-To` and `References` headers.
4.  **Match**: 
    - If the reply references a `last_message_id` stored in our `rfp_invites` table, it is a **100% matched reply**.
    - If no header match is found, it attempts to match using the **Carrier's Email** and **RFP Title** in the subject.
5.  **Record**: The email content is stored in the `inbound_emails` table and linked to the correct `rfp_id` and `carrier_id`.
6.  **Cleanup**: The email is marked as "Read" in Gmail so it isn't processed twice.

**Key Files:**
- `src/services/gmailService.ts`: Gmail API client and parsing logic.
- `src/app/api/email/fetch-gmail/route.ts`: The sync worker endpoint.

---

## 4. Database Schema

The following tables handle the email logic:

- **`rfp_invites`**: Stores `last_message_id` for tracking.
- **`inbound_emails`**: Stores the content of received emails.
- **`carriers`**: Used to match the sender's email address to a known user.

---

## 5. Maintenance & Syncing

To keep the dashboard updated with new replies, you can:
1.  **Manually Sync**: Visit `http://localhost:3000/api/email/fetch-gmail` in your browser.
2.  **Automate**: Set up a "Cron Job" or a repeating task to hit the sync endpoint every few minutes.
